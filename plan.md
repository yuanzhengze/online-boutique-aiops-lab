# 10 天开发计划

## 1. 项目目标

本项目计划在 10 天内完成软件测试与维护大作业的核心开发、实验、文档和展示准备工作。

项目目标包括：

- 完成 Online Boutique 微服务系统部署。
- 完成 Prometheus、Grafana、ChaosMesh 监控与故障注入链路。
- 新增 Review Service 商品评价服务，满足第三档微服务开发要求。
- 实现 Grafana 智能分析与邮件通知 Agent 模块。
- 使用 Selenium 和 JMeter 完成功能测试与性能测试。
- 复现并对比至少 4 篇异常检测或故障诊断相关论文方法。
- 完成 GitHub 仓库、PDF 报告、展示 PPT 和展示排练。

## 2. 成员职责概览

- 袁正泽：项目选题、服务器环境搭建、Online Boutique 初始部署、项目规则制定、项目统筹规划。
- 黄开轩：核心微服务开发、Review Service 设计与实现、复杂系统集成、Agent 工程实现协助。
- 陈宇轩：论文复现负责人，负责算法实验框架、核心论文复现、多算法对比分析。
- 陈文涛：Prometheus、Grafana、数据采集、Agent 模块牵头实现。
- 谭张锐：ChaosMesh 故障注入、JMeter 性能测试、故障场景数据构造。
- 傅昱翔：Selenium 功能测试、新增服务功能验证、轻量论文复现辅助、测试材料整理。

### Day 6（6.5）：故障注入与性能测试

主要目标：

- 部署并验证 ChaosMesh。
- 设计至少 3 类故障注入场景。
- 使用 JMeter 构造正常负载和故障负载。
- 采集故障期间的监控数据。

负责人：

- 谭张锐：ChaosMesh 和 JMeter 主负责人。
- 陈文涛：配合采集故障监控数据。

建议故障场景：

- Pod Kill。
- Network Delay。
- Network Loss。
- CPU Stress。
- Memory Stress。

交付物：

- ChaosMesh 实验配置。
- JMeter 测试计划。
- 故障注入记录表。
- 性能测试结果。
- 故障场景监控数据。

验收标准：

- 至少 3 类故障注入成功。
- JMeter 能输出响应时间、吞吐量和错误率。
- 故障数据可用于论文算法和 Agent 模块分析。

负责人：

交付物：

- Agent 模块源码或脚本。

- 邮件通知截图。
- 正常场景与故障场景分析结果。

验收标准：

- Agent 能获取监控数据。
- Agent 能生成分析摘要。
- Agent 能发送邮件。

- 统一评估指标并生成对比结果。

负责人：

- 陈宇轩：论文复现和算法对比主负责人。

验收标准：

- 至少 4 种论文方法或算法思路有对比结果。
- 结果包含 Precision、Recall、F1、AUC 或其他合理指标。

## 4. 每日同步机制

每天进行一次短会或线上同步，内容包括：

- 昨天完成了什么。
- 今天计划完成什么。
- 当前遇到什么阻塞。
- 是否需要其他成员协助。
- 当天是否有可提交的截图、脚本、配置或实验结果。

袁正泽负责维护每日进度表，所有成员每天至少提交一次阶段性产物。

## 5. 最低可交付版本

如果时间紧张，优先保证以下最低可交付版本：

- Online Boutique 可运行。
- Review Service 可独立部署并通过 API 展示。
- Prometheus 和 Grafana 可展示核心指标。
- ChaosMesh 至少完成 3 类故障注入。
- Selenium 和 JMeter 至少各有一套可运行脚本。
- Agent 完成“获取数据、生成摘要、发送邮件”闭环。
- 至少 4 种论文方法或算法思路有对比结果。
- 报告和 PPT 能完整说明分工、过程、结果和加分项。

## 6. 风险控制

- 如果 Review Service 前端集成受阻，保留 API 调用和 Kubernetes 部署展示作为兜底。
- 如果 Grafana API 调用受阻，Agent 可改用 Prometheus 查询接口或已导出的监控数据。
- 如果论文完整复现难度过高，采用“重点复现 2 篇 + 轻量对比 2 篇”的策略。
- 如果部署环境不稳定，优先保留截图、日志和录屏作为展示材料。
- 如果报告和 PPT 时间不足，各模块负责人必须先提交本模块最小可用内容，再统一润色。
