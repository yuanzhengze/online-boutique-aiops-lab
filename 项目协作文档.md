## 4. GitHub 协作规则

### 提交原则

- 每个人只提交自己负责模块相关内容。
- 不随意修改他人负责的代码或文档。
- 如果必须修改他人模块，先在群里说明原因。
- 每次提交尽量小而清晰，避免一次提交大量无关文件。
- 不提交无关临时文件、缓存文件、压缩包或个人环境配置。

### Commit 信息规范

建议使用以下格式：

```text
类型: 简短说明
```

常用类型：

- `deploy`：部署相关。
- `service`：微服务开发相关。
- `monitor`：Prometheus、Grafana、数据采集相关。
- `agent`：Agent 模块相关。
- `test`：Selenium、JMeter 测试相关。
- `chaos`：ChaosMesh 故障注入相关。
- `algo`：论文复现和算法实验相关。
- `docs`：报告、PPT、README 等文档相关。

示例：

```text
deploy: add online boutique minikube deployment notes
service: implement review service basic APIs
agent: add grafana metrics email report script
docs: update member contribution section
```

## 9. Agent 模块协作说明

Agent 模块目标是实现一个轻量闭环：

```text
Grafana/Prometheus 获取指标 -> 分析关键指标 -> 生成摘要 -> 发送邮件
```

优先实现的功能：

- 获取 CPU、内存、请求延迟、错误率、吞吐量等指标。
- 基于规则判断是否异常。
- 生成简短分析报告。
- 通过邮件发送结果。

Agent 模块不需要做得过重，重点是展示“监控数据自动分析和通知”的流程。

如果 Grafana API 调用不稳定，可以使用 Prometheus 查询接口或导出的监控数据作为输入。
