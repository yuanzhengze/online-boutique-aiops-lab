# 软件测试与维护大作业任务规划与分工

## 1. 项目定位

本组选择完成大作业评分标准中的**第三档**，并以**加分项一**为主要加分目标。同时，为了提高展示完整度，额外实现一个轻量级智能运维 Agent 模块：从 Grafana 获取监控信息，整理分析系统状态，并通过邮件发送分析结果。

第三档要求为：在第二档基础上，选择更复杂的开源微服务系统进行部署、监控和维护，并完成该项目中一到两个微服务开发。

加分项一要求为：在基础复现两篇异常检测与故障诊断相关论文的基础上，进一步选择更多论文进行复现与对比。

智能运维 Agent 模块可辅助覆盖监控数据读取、异常信息整理、运维分析和结果通知等内容，作为项目中独立但轻量的增强模块进行展示。

## 2. 项目选型

建议选择 **Online Boutique** 作为本次大作业的微服务系统。

选择理由：

- 相比 SockShop，Online Boutique 服务数量更多，系统结构更复杂，符合第二档基础要求。
- 项目部署资料较完整，适合在 Minikube 和 Kubernetes 环境中运行。
- 服务边界清晰，便于进行 Prometheus 监控、Grafana 可视化和 ChaosMesh 故障注入。
- 前端、订单、购物车、商品、支付等业务流程完整，适合进行 Selenium 功能测试和 JMeter 性能测试。
- 具备扩展空间，适合新增商品评价、优惠券等微服务，以满足第三档要求。

## 3. 微服务开发目标

本项目计划新增 1 到 2 个微服务：

1. **Review Service 商品评价服务**
   - 作为核心开发任务。
   - 支持用户查看商品评价、提交评价、查询商品评分。
   - 与商品详情页或前端服务进行交互。
   - 可接入 Prometheus 指标采集，展示新增服务的运行状态。

2. **Coupon Service 优惠券服务**
   - 作为可选冲刺任务。
   - 支持优惠券查询、使用和折扣计算。
   - 可与购物车或结算流程进行简单集成。

优先保证 Review Service 完整落地。如果时间充足，再完成 Coupon Service。

## 4. 智能运维 Agent 模块目标

本项目新增一个独立的轻量级智能运维 Agent 模块，用于从 Grafana 获取系统监控信息，整理分析关键指标，并将分析结果通过邮件发送给项目成员或运维负责人。

该模块不追求复杂的自主决策，而是优先保证功能闭环清晰、实现成本可控、展示效果明确。

核心功能：

- 通过 Grafana API 或 Prometheus 查询接口获取监控数据。
- 提取 CPU、内存、请求延迟、错误率、吞吐量等关键指标。
- 根据预设规则判断系统是否存在异常趋势，例如延迟升高、错误率增加、资源使用率过高。
- 自动生成简短运维分析摘要，包括当前状态、异常指标、可能影响和建议处理方向。
- 通过邮件发送分析报告。

模块价值：

- 与 Prometheus 和 Grafana 监控任务直接衔接。
- 可使用 ChaosMesh 故障注入数据验证分析结果。
- 可作为智能运维方向的轻量展示模块。
- 能够丰富最终展示中的“监控、维护、异常分析、自动通知”链路。

## 5. 总体交付目标

最终项目需要完成以下内容：

- 部署 Online Boutique 至本地 Minikube 集群。
- 配置 Prometheus 采集微服务系统指标。
- 配置 Grafana 展示监控数据。
- 部署 ChaosMesh，并完成多种故障注入实验。
- 使用 Selenium 完成功能测试。
- 使用 JMeter 完成性能测试。
- 新增至少 1 个微服务，并完成部署、调用和展示。
- 实现 Grafana 智能分析与邮件通知 Agent 模块。
- 复现至少 4 篇异常检测或故障诊断相关论文。
- 对不同论文方法进行实验对比和结果分析。
- 完成 GitHub 开源仓库、PDF 报告和展示 PPT。

## 11. 风险与应对

### 风险一：复杂微服务系统部署困难

应对方案：

- 优先保证 Online Boutique 原系统部署成功。
- 保留 SockShop 作为兜底参考，但最终目标仍以 Online Boutique 为主。
- 袁正泽每日检查部署风险，黄开轩及时记录部署问题和解决方法。

### 风险二：新增微服务与原系统集成成本高

应对方案：

- 优先实现独立可访问的 Review Service。
- 再逐步接入前端或商品详情页。
- 如果深度集成受阻，至少保证新增服务具备 API 调用、K8s 部署和展示能力。

### 风险三：论文复现难度过高

应对方案：

- 陈宇轩优先选择有开源代码或实现资料的论文。
- 将论文分为“重点复现”和“轻量对比”两类。
- 傅昱翔辅助完成较轻量论文，降低陈宇轩压力。

### 风险四：数据质量不足

应对方案：

- 陈文涛和谭张锐提前统一故障注入记录格式。
- 每次实验记录故障类型、开始时间、结束时间、影响服务和采集指标。
- 算法实验前先完成一轮数据可用性检查。

### 风险五：Agent 模块联调时间不足

应对方案：

- Agent 模块优先保证“读取监控数据、生成摘要、发送邮件”三步闭环。
- 分析逻辑采用规则判断，不引入过重的模型或复杂智能体框架。
- 如果 Grafana API 调用受阻，可直接使用 Prometheus 查询接口或导出的监控数据作为输入。
- 邮件发送优先使用脚本方式实现，展示时保留运行日志和邮件截图。

- 袁正泽从第 1 天开始维护项目结构和最终材料清单。
